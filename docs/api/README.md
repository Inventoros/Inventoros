# Inventoros REST API

This directory contains the API reference for Inventoros.

- **`openapi.yaml`** — OpenAPI 3.0 specification covering every public REST endpoint.
- **`README.md`** (this file) — narrative overview, auth, rate limits, error envelope, code examples.

The full v1 surface is mounted under `/api/v1`. There is also a separate (auth-required) Sanctum-protected GraphQL endpoint at `POST /graphql` powered by `rebing/graphql-laravel`; it exposes products, orders, suppliers, purchase orders, stock adjustments, locations and categories. This document focuses on the REST API.

A live, interactive Stoplight Elements UI is also available at `/docs/api`, generated by `dedoc/scramble` from the controller doc-blocks.

## Base URL

```
{APP_URL}/api/v1
```

where `APP_URL` matches the `APP_URL` value in your `.env` (e.g. `http://localhost`, `https://inventoros.example.com`).

## Authentication

Inventoros uses [Laravel Sanctum](https://laravel.com/docs/sanctum) personal-access tokens. Every request (except `POST /api/v1/login`) must include:

```
Authorization: Bearer <your-token>
```

### Obtain a token

```bash
curl -X POST "${APP_URL}/api/v1/login" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -d '{
    "email": "you@example.com",
    "password": "your-password",
    "device_name": "my-laptop"
  }'
```

Successful response (`200`):

```json
{
  "message": "Login successful",
  "user": {
    "id": 1,
    "name": "Jane Doe",
    "email": "you@example.com",
    "role": "admin",
    "organization_id": 1
  },
  "token": "1|aBc...XYZ",
  "token_type": "Bearer"
}
```

Store `token` securely. The plain-text value is shown **only once**.

### Token management

| Endpoint                              | Purpose                                            |
| ------------------------------------- | -------------------------------------------------- |
| `POST   /api/v1/login`                | Issue a new token (rate-limited 5/min/IP).         |
| `POST   /api/v1/logout`               | Revoke the current token.                          |
| `GET    /api/v1/user`                 | Get the authenticated user + permissions.          |
| `POST   /api/v1/tokens`               | Create a named token with optional ability scopes. |
| `DELETE /api/v1/tokens/{tokenId}`     | Revoke a specific token by id.                     |

## Multi-tenancy

Every record carries an `organization_id` and is scoped to the authenticated user's organization. Cross-organization access returns `404 not_found`, **not** `403`, to avoid leaking the existence of other organizations' resources.

## Permissions

Almost every route is guarded by the `api.permission:` middleware (alias of `App\Http\Middleware\CheckApiPermission`) which checks the authenticated user against one or more permission strings. The middleware accepts `|`-separated permissions and grants access when the user holds **any** of them (the default), e.g.:

```
api.permission:view_products|manage_products
```

Common permission strings include `view_products`, `manage_products`, `view_orders`, `manage_orders`, `view_suppliers`, `manage_suppliers`, `view_purchase_orders`, `manage_purchase_orders`, `receive_purchase_orders`, `edit_purchase_orders`, `view_categories`, `manage_categories`, `view_locations`, `manage_locations`, `view_warehouses`, `manage_warehouses`, `view_stock_audits`, `manage_stock_audits`, `view_stock_adjustments`, `manage_stock`, `view_roles`, `manage_roles`, `view_reports`.

A request lacking the required permission returns `403 forbidden`.

## Rate limiting

| Scope                      | Limit             | Keyed by             | Defined in                                  |
| -------------------------- | ----------------- | -------------------- | ------------------------------------------- |
| Entire `/api/v1/*` group   | **60 req/min**    | user id (or IP)      | `App\Providers\AppServiceProvider::boot()`  |
| `POST /api/v1/login`       | **5 req/min**     | IP                   | inline `throttle:5,1` in `routes/api.php`   |

When a request exceeds the limit you'll get `429 Too Many Requests` with a `Retry-After` header (in seconds). Successful responses include the standard Laravel rate-limit headers `X-RateLimit-Limit` and `X-RateLimit-Remaining`.

Tip: a single `429` from `POST /login` is keyed by IP, so don't spin up a parallel test loop against `/login` from one machine — rotate users via `POST /tokens` once you have an initial token.

## Pagination

List endpoints (those backed by `AnonymousResourceCollection`) return Laravel's standard envelope:

```json
{
  "data":  [ ... ],
  "links": { "first": "...", "last": "...", "prev": null, "next": "..." },
  "meta":  {
    "current_page": 1,
    "from": 1,
    "last_page": 3,
    "path": "...",
    "per_page": 15,
    "to": 15,
    "total": 42
  }
}
```

Two query parameters apply uniformly:

- `per_page` — items per page (default `15`, max **`100`**)
- `page` — page number (1-indexed)

Most list endpoints also expose:

- `search` — free-text search against the most relevant fields
- `sort_by`, `sort_dir` — sort by a whitelisted column, `asc` or `desc` (default `desc`)

Single-item endpoints (`show`, `store`, `update`) return:

```json
{ "message": "...", "data": { ... } }
```

(`message` is omitted for plain `GET show` responses.)

## Error envelope

Every error response is JSON. The shape is consistent across the API:

```json
{
  "message": "Human-readable summary",
  "error":   "machine_readable_code"
}
```

Validation errors additionally include the standard Laravel `errors` map:

```json
{
  "message": "The given data was invalid.",
  "errors": {
    "email": ["The email field is required."],
    "items.0.product_id": ["The selected product id is invalid."]
  }
}
```

| HTTP status | `error` codes you'll see                                             | When                                                                         |
| ----------- | -------------------------------------------------------------------- | ---------------------------------------------------------------------------- |
| `401`       | `unauthenticated`                                                    | Missing/invalid Bearer token.                                                |
| `403`       | `forbidden`                                                          | Authenticated but lacks required permission.                                 |
| `404`       | `not_found`                                                          | Resource doesn't exist or belongs to another organization.                   |
| `422`       | `insufficient_stock`, `cannot_edit`, `cannot_send`, `cannot_delete`, `cannot_receive`, `cannot_cancel`, `no_items_received`, `has_products`, `cannot_delete_default`, `option_in_use`, `max_options_exceeded`, `duplicate_option`, `invalid_product_type`, `circular_reference`, `no_components`, `invalid_status`, `invalid_data_source`, `invalid_columns` | Domain rule violation (in addition to standard validation errors). |
| `429`       | (no `error` field; see `Retry-After` header)                         | Rate limit exceeded.                                                         |
| `500`       | `query_failed`                                                       | Saved-report execution failure.                                              |

## Resource overview

All paths are relative to `/api/v1`.

| Resource              | Routes                                                                                          |
| --------------------- | ----------------------------------------------------------------------------------------------- |
| Auth                  | `POST /login`, `POST /logout`, `GET /user`, `POST /tokens`, `DELETE /tokens/{id}`                |
| Products              | `GET/POST /products`, `GET/PUT/DELETE /products/{id}`                                            |
| Product Options       | `GET/POST /products/{p}/options`, `POST /products/{p}/options/reorder`, `GET/PUT/DELETE /products/{p}/options/{id}` |
| Product Variants      | `GET/POST /products/{p}/variants`, `POST /products/{p}/variants/bulk`, `GET/PUT/DELETE /products/{p}/variants/{id}`, `POST /products/{p}/variants/{id}/adjust-stock` |
| Product Components    | `GET/POST /products/{p}/components`, `PUT/DELETE /products/{p}/components/{id}`                  |
| Batch Tracking        | `GET/POST /products/{p}/batches`, `GET /products/{p}/batches/{id}`                               |
| Serial Tracking       | `GET/POST /products/{p}/serials`, `GET/PUT /products/{p}/serials/{id}`                           |
| Categories            | `GET/POST /categories`, `GET/PUT/DELETE /categories/{id}`                                        |
| Locations             | `GET/POST /locations`, `GET/PUT/DELETE /locations/{id}`                                          |
| Warehouses            | `GET/POST /warehouses`, `GET/PUT/DELETE /warehouses/{id}`                                        |
| Orders                | `GET/POST /orders`, `GET/PUT/DELETE /orders/{id}`                                                |
| Stock Adjustments     | `GET/POST /stock-adjustments`, `GET /stock-adjustments/{id}`                                     |
| Stock Audits          | `GET /stock-audits`, `GET /stock-audits/{id}` (read-only)                                        |
| Suppliers             | `GET/POST /suppliers`, `GET/PUT/DELETE /suppliers/{id}`                                          |

| Permission Sets       | `GET/POST /permission-sets`, `GET /permission-sets/categories`, `GET/PUT/DELETE /permission-sets/{id}` |
| Saved Reports         | `GET/POST /reports`, `GET/PUT/DELETE /reports/{id}`, `GET /reports/{id}/export` (CSV)            |

See `openapi.yaml` for full request/response schemas and parameters.

## Code examples

The examples below assume:

```bash
APP_URL="http://localhost"
TOKEN="1|paste-your-token-here"
```

### curl

#### List low-stock products

```bash
curl -G "${APP_URL}/api/v1/products" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Accept: application/json" \
  --data-urlencode "low_stock=1" \
  --data-urlencode "per_page=25"
```

#### Create a product

```bash
curl -X POST "${APP_URL}/api/v1/products" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -H "Accept: application/json" \
  -d '{
    "sku": "WIDGET-001",
    "name": "Blue Widget",
    "selling_price": 19.99,
    "purchase_price": 8.50,
    "stock": 100,
    "min_stock": 10,
    "tracking_type": "none"
  }'
```

#### Adjust stock

```bash
curl -X POST "${APP_URL}/api/v1/stock-adjustments" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "product_id": 42,
    "quantity": -3,
    "type": "damage",
    "reason": "Dropped pallet",
    "notes": "Shrink-wrap on box 14 was torn."
  }'
```

#### Create an order

```bash
curl -X POST "${APP_URL}/api/v1/orders" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Content-Type: application/json" \
  -d '{
    "customer_name": "Acme Corp",
    "customer_email": "ap@acme.example",
    "currency": "USD",
    "items": [
      { "product_id": 42, "quantity": 2 },
      { "product_id": 99, "quantity": 1, "unit_price": 49.95 }
    ]
  }'
```

#### Look up a product by barcode

```bash
curl "${APP_URL}/api/v1/barcode/0123456789012" \
  -H "Authorization: Bearer ${TOKEN}" \
  -H "Accept: application/json"
```

### PHP (using `Http` facade)

Inventoros itself is a Laravel app, so this is the most natural client style:

```php
use Illuminate\Support\Facades\Http;

$client = Http::baseUrl(config('services.inventoros.url').'/api/v1')
    ->withToken(config('services.inventoros.token'))
    ->acceptJson();

// List products
$products = $client->get('products', [
    'search'   => 'widget',
    'per_page' => 25,
])->throw()->json('data');

// Create an order
$order = $client->post('orders', [
    'customer_name' => 'Acme Corp',
    'currency'      => 'USD',
    'items' => [
        ['product_id' => 42, 'quantity' => 2],
    ],
])->throw()->json('data');

// Adjust stock
Http::baseUrl(config('services.inventoros.url').'/api/v1')
    ->withToken(config('services.inventoros.token'))
    ->acceptJson()
    ->post('stock-adjustments', [
        'product_id' => 42,
        'quantity'   => -3,
        'type'       => 'damage',
        'reason'     => 'Dropped pallet',
    ])
    ->throw();
```

`->throw()` raises on `4xx`/`5xx`; wrap calls in `try { ... } catch (\Illuminate\Http\Client\RequestException $e) { ... }` and inspect `$e->response->json()` to read the error envelope.

### JavaScript (browser / Node `fetch`)

```js
const APP_URL = "http://localhost";
const TOKEN   = "1|paste-your-token-here";

const api = (path, init = {}) =>
  fetch(`${APP_URL}/api/v1${path}`, {
    ...init,
    headers: {
      Accept: "application/json",
      "Content-Type": "application/json",
      Authorization: `Bearer ${TOKEN}`,
      ...(init.headers || {}),
    },
  }).then(async (r) => {
    const body = await r.json().catch(() => ({}));
    if (!r.ok) {
      const err = new Error(body.message || `HTTP ${r.status}`);
      err.status = r.status;
      err.body = body;
      throw err;
    }
    return body;
  });

// List products
const { data: products } = await api("/products?low_stock=1&per_page=25");

// Create an order
const { data: order } = await api("/orders", {
  method: "POST",
  body: JSON.stringify({
    customer_name: "Acme Corp",
    currency: "USD",
    items: [{ product_id: 42, quantity: 2 }],
  }),
});

// Adjust stock
await api("/stock-adjustments", {
  method: "POST",
  body: JSON.stringify({
    product_id: 42,
    quantity: -3,
    type: "damage",
    reason: "Dropped pallet",
  }),
});
```

For Vue/Inertia frontends inside Inventoros itself, the same token mechanism works — you can hit `/api/v1` with an `Authorization` header and the user's session will not be used.

## Webhook events (note)

Webhook event subscription (`WebhookEventSubscriber`) is wired up in `AppServiceProvider`, but at the time of writing **there are no public REST endpoints to manage webhook configurations** (this is listed as "Webhook System — NOT STARTED" in `todo.txt`). Webhooks fire from internal events only. This will be documented here once management endpoints land.

## Versioning

The current API is `v1`, served at `/api/v1`. Future incompatible changes will be released under `/api/v2`; `v1` will continue to receive non-breaking additions.
